## Phase 3：Architecture 关键发现

### Gateway → H2 Proxy 路由机制

老环境 (`openclaw-multitenancy`, us-east-1) 的工作原理：
- 安装了两个包：`openclaw-agentcore@1.0.3` + `openclaw@2026.3.24`
- Gateway 进程用的是 `openclaw-agentcore` 的 binary（不是 `openclaw`）
- `openclaw-agentcore` 的 Gateway 尊重 `baseUrl: http://localhost:8091`
- H2 Proxy 在端口 8091 监听，拦截所有 Bedrock 调用

新环境 (`openclaw-jiade2`, ap-northeast-1) 的问题：
- 只安装了 `openclaw@2026.3.24`
- `openclaw@2026.3.24` 的 Gateway 忽略 `baseUrl` 配置
- 解决方案：`AWS_ENDPOINT_URL_BEDROCK_RUNTIME=http://localhost:8091` 环境变量
- 这是 AWS SDK v3 的标准 endpoint override，Gateway 无法绕过

### 完整数据流（已验证）

```
飞书/Telegram/Discord 消息
  → EC2 Gateway (port 18789, openclaw@2026.3.24)
  → Gateway agent 调 Bedrock API
  → AWS SDK 被 AWS_ENDPOINT_URL_BEDROCK_RUNTIME 重定向到 localhost:8091
  → H2 Proxy (port 8091)
     - 提取 channel + userId 从 Bedrock request metadata
     - PATH C: 检查 /start TOKEN → pairing flow
     - Binding check: DynamoDB MAPPING# 查询
     - 路由到 Tenant Router
  → Tenant Router (port 8090)
     - MAPPING# 解析: feishu OU ID → emp-mike
     - 查 DDB 获取 positionId
     - 调用 AgentCore Runtime
  → AgentCore microVM
     - workspace_assembler.py 从 DDB/S3 组装 3 层 SOUL
     - server.py 运行 openclaw agent CLI
     - 响应原路返回
```

### IM 员工自助 Pairing 流程（已验证）

```
1. Admin 在 Gateway UI 配置飞书 bot (一次性, DM policy = open)
2. 员工登录 Portal → Connect IM → Feishu → 获取 pairing token
3. 员工在飞书给 bot 发 /start TOKEN
4. H2 Proxy PATH C 拦截 → 调 pair-pending API → 返回确认提示
5. 员工回复 BIND
6. H2 Proxy 调 pair-complete API → DynamoDB 写入 MAPPING#
7. 之后每条飞书消息 → H2 Proxy 查到 MAPPING → 路由到员工的 AgentCore VM
```
